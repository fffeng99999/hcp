# 3.3 数据存储设计

本平台将数据分为**区块链数据**与**系统数据**两类，分别由 HCP-Consensus 和 HCP-Lab/HCP-Loadgen 子系统维护，如图 3-1 所示。

**区块链数据**包括交易记录、区块状态和验证者信息，由 HCP-Consensus 通过 RocksDB 引擎与 IAVL+Tree 键值存储进行管理，用于支持共识验证和历史追溯；**系统数据**包括实验配置、节点状态、性能指标和日志，由 HCP-Loadgen 通过 PostgreSQL 数据库进行结构化存储，以及 HCP-Lab 通过 JSON/CSV 文件进行实验结果记录，用于实验自动化管理和统计分析。

这种数据分层设计具有两方面的意义：一方面，区块链数据按照共识系统的状态管理逻辑独立维护，保证交易提交和状态转移过程具有可验证性；另一方面，系统数据不参与链上共识决策，而是作为实验观测数据独立采集，从而避免性能监控过程对共识核心路径产生额外干扰。基于此设计，平台能够在统一环境中同时完成共识运行、负载注入、指标采集和结果分析，为第四章的基准对比实验、参数扫描实验和消融实验提供可复现的数据基础。

---

## 3.3.1 区块链数据存储

区块链数据的底层存储采用 **RocksDB 引擎**，并在此基础上构建 **IAVL+Tree 版本化键值存储**，形成"共识层 → 应用层 → IAVL+Tree → RocksDB"的四层架构（图 3-2）。该架构负责持久化以下五类核心数据：

**(1) 交易数据**：由 HCP-Loadgen 构造并提交至共识引擎的交易集合，包含订单信息、账户信息和时间戳等字段，经 Mempool 排序后随区块提交至链上。

**(2) 区块数据**：已完成共识确认的交易集合及对应的提交证明（Commit Proof），支持历史交易追溯和共识延迟分析。

**(3) 应用状态数据**：交易执行后的账户余额、订单状态等应用层状态，通过 IAVL+Tree 进行版本化管理，支持快照查询和状态恢复。

**(4) 验证者集合信息**：当前参与共识的节点身份、公钥和投票权重，在 tPBFT 等算法中还会受到信任评分机制影响。

**(5) 信任评分状态**：tPBFT 算法中各节点的信任分数、成功率和响应时间等指标，采用 100 轮滑动窗口维护，用于动态验证者筛选。

> ⚠️ **设计原则**：平台仅保留与区块提交、状态恢复、信任评分更新和实验分析相关的必要数据，避免过度持久化对共识性能产生额外开销。

## 3.3.2 系统数据存储

系统数据采用**双引擎架构**，分别服务于不同的实验需求（图 3-3）：

### (1) PostgreSQL 结构化存储（HCP-Loadgen）

HCP-Loadgen 使用 PostgreSQL 存储负载生成的业务数据，通过 **Schema 隔离机制**确保不同实验之间的数据独立性。每次实验启动时自动创建独立 Schema（如 `exp1_tpbft`），包含以下四类表：

| 表名 | 用途 | 关键字段 |
|------|------|----------|
| accounts | 账户信息 | account_id, address |
| balances | 余额快照 | account_id, asset_symbol, available, frozen |
| orders | 订单信息 | order_id, side, price, quantity, status |
| trades | 交易记录 | trade_id, tx_hash, latency_ms |

该存储方案采用连接池管理、批量写入和 UPSERT 语义等技术优化，确保高频负载注入场景下的数据写入效率。

### (2) 文件系统存储（HCP-Lab）

HCP-Lab 将实验结果按"实验编号 → 算法类型 → 节点规模"的三级目录结构保存到文件系统：

```
report/
├── exp{N}_{algorithm}/          # 按实验编号和算法类型组织
│   ├── nodes_{count}/            # 按节点规模分组
│   │   ├── result.json           # 性能指标（TPS/P50/P95/P99/CPU/Memory）
│   │   ├── metrics.csv           # 详细时序数据
│   │   └── logs/                 # 运行日志（区块提交/交易确认/RocksDB写入）
│   └── figures/                  # 图表输出（PNG/SVG）
└── summary.json                  # 实验总览
```

JSON 格式的 result.json 记录每轮实验的核心性能指标（吞吐量、延迟分位数、资源利用率、存储性能），CSV 格式的 metrics.csv 提供详细的时序数据供后续统计分析使用。

### (3) 四类系统数据职责

| 数据类别 | 维护模块 | 用途 |
|----------|----------|------|
| **实验配置** | HCP-Lab 参数矩阵 | 记录节点规模、交易数量、算法类型、随机种子等参数，保证实验条件可复现 |
| **节点运行状态** | SystemMonitor | 实时采集 CPU 利用率、内存占用、网络带宽和磁盘 I/O，用于瓶颈定位 |
| **性能指标** | HCP-Lab 统计模块 | 基于交易发送时间和确认回执时间计算 TPS、P99 延迟等关键指标，是第四章实验的主要数据来源 |
| **日志追踪** | 日志解析器 | 解析区块提交、交易确认、RocksDB 写入和共识失败等事件日志，支撑异常分析和根因排查 |

---

## 小结

本章的数据存储设计遵循**"区块链数据服务共识、系统数据服务实验"**的原则。区块链数据通过 RocksDB + IAVL+Tree 保证链上状态的完整性和可追溯性；系统数据通过 PostgreSQL 和文件系统实现实验过程的全面记录和分析能力。二者在功能上相互分离，共同为第四章的性能评估实验提供坚实的数据基础。
