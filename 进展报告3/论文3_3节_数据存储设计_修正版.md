# 3.3 数据存储设计（修正版）

为了保证 HCP-Bench 平台既能够稳定运行区块链共识过程，又能够完整记录实验过程中的性能数据，本文将平台数据划分为**区块链数据**与**系统数据**两类。

- **区块链数据**：主要由 HCP-Consensus 子系统维护，直接服务于交易执行、区块提交、状态更新与共识结果验证；
- **系统数据**：主要由 HCP-Loadgen 与 HCP-Lab 子系统产生和管理，用于支撑实验参数配置、运行过程监控、性能指标统计和结果报告生成。

二者在功能上相互关联，但在职责上保持分离：区块链数据关注链上状态的一致性和可追溯性，系统数据关注实验过程的可复现性和性能分析的完整性。

这种数据划分方式有助于降低实验系统与共识执行路径之间的耦合。一方面，区块链数据按照区块链系统自身的状态管理逻辑进行维护，保证交易提交和状态转移过程具有可验证性；另一方面，系统数据不参与链上共识决策，而是作为实验观测数据独立记录，从而避免性能采集与日志分析过程对共识核心路径产生额外干扰。基于上述设计，HCP-Bench 能够在统一平台中同时完成共识运行、负载注入、指标采集和结果分析，为第四章的基准对比实验、参数扫描实验和消融实验提供数据基础。

---

## 3.3.1 区块链数据存储设计

区块链数据主要指共识引擎在交易排序、区块生成、状态提交和历史追溯过程中需要维护的数据。在本文平台中，该类数据主要包括**交易数据、区块数据、应用状态数据、验证者集合信息以及 tPBFT 相关的信任评分状态**。

### 存储技术栈

区块链数据的底层存储采用 **RocksDB 引擎**（配置项 `db_backend = "rocksdb"`），并在此基础上构建 **IAVL+Tree KV Store**（配置项 `StoreType = IAVL`），形成如图 3-2 所示的四层存储架构：

```
┌─────────────────────────────────────────┐
│  第一层：HCP-Consensus 共识子系统        │
│  (tPBFT / Hierarchical-tPBFT / HotStuff / Raft)
├─────────────────────────────────────────┤
│  第二层：Cosmos SDK 应用层 (ABCI)       │
│  BeginBlock() → DeliverTx() → EndBlock() → Commit()
├─────────────────────────────────────────┤
│  第三层：IAVL+Tree KV Store              │
│  (版本化键值存储，缓存大小: 781250 nodes) │
├─────────────────────────────────────────┤
│  第四层：RocksDB 存储引擎 (LSM-Tree)    │
│  WAL → MemTable → SSTable → Compaction  │
└─────────────────────────────────────────┘
           ↓
┌─────────────────────────────────────────┐
│  物理磁盘 (data/ 目录)                  │
│  block_store.db | state.db | application.db | wal/
└─────────────────────────────────────────┘
```

### 数据类型详解

#### 1. 交易数据 (Transaction Data)

交易数据由 HCP-Loadgen 构造并通过 Broadcaster 提交至共识引擎。每笔交易包含以下字段：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| order_id | BIGINT | 订单编号 |
| account_id | BIGINT | 账户编号 |
| side | VARCHAR(10) | 交易方向 (BUY/SELL) |
| price | DECIMAL(20,8) | 价格 |
| quantity | DECIMAL(20,8) | 数量 |
| timestamp | TIMESTAMP | 时间戳 |
| nonce | BIGINT | 随机数（防重放） |

交易进入共识引擎后，首先进入待处理队列（Mempool），并在共识过程中被排序、广播和打包，最终随区块提交至链上。

#### 2. 区块数据 (Block Data)

区块数据用于记录已经完成共识确认的交易集合，主要包含：

```go
type BlockData struct {
    Height          int64         // 区块高度
    Timestamp       time.Time     // 区块时间戳
    Txs             [][]byte      // 交易列表
    Hash            []byte        // 区块哈希
    PrevBlockHash   []byte        // 前一区块哈希
    CommitProof     *CommitProof  // 提交证明（PBFT/tPBFT必需）
}
```

区块数据的保存使系统能够对历史交易进行追溯，并支持后续实验中对**区块确认时间、交易提交顺序和共识延迟**的分析。对于 PBFT、CometBFT 和 tPBFT 等多轮投票型共识算法而言，**区块提交证明 (Commit Proof)** 是验证共识结果有效性的重要依据，因此平台需要保留与最终提交结果相关的必要证明信息。

#### 3. 应用状态数据 (Application State Data)

应用状态数据由 Cosmos SDK 应用层在交易执行后更新，主要通过 IAVL+Tree 进行版本化持久化管理：

```go
// 挂载 IAVL Store
for _, key := range keys {
    app.MountStore(key, storetypes.StoreTypeIAVL)
}
```

记录的内容包括：
- **账户余额** (account_balance)：各账户的可用余额和冻结余额
- **交易状态** (trade_status)：订单的成交状态（PENDING/FILLED/CANCELLED）
- **应用层键值状态** (kv_state[])：模块自定义的状态数据

IAVL+Tree 的优势在于支持**版本快照查询**，保证节点在重启或恢复时能够重新加载已提交状态。对于高频交易模拟场景而言，应用状态数据不仅用于验证交易执行结果，也为后续统计**交易成功率、冲突率和确认状态**提供基础。

#### 4. 验证者集合信息 (Validator Set Information)

验证者集合信息用于记录当前参与共识的节点集合及其身份信息：

```go
type ValidatorSet struct {
    Validators []*Validator // 验证者列表
}

type Validator struct {
    Address     sdk.ValAddress // 验证者地址
    PubKey      crypto.PubKey  // 公钥
    VotingPower int64          // 投票权重
}
```

在 tPBFT 系列算法中，验证者集合会受到信任评分机制影响，因此平台还需要维护节点信任值、历史响应情况、成功率和筛选结果等辅助状态。

#### 5. 信任评分状态 (Trust Scoring State)

信任评分状态由 [trust_scorer.go](consensus/tpbft/trust_scorer.go) 中的 `TrustScorer` 结构体维护：

```go
type TrustScorer struct {
    scores      map[string]*TrustScore  // 节点信任分数映射
    historySize int                      // 历史窗口大小 = 100轮
}

type TrustScore struct {
    SuccessRate   float64 // 历史成功率
    ResponseTime  float64 // 平均响应时间
    StakingWeight float64 // 质押权重
    History       []RoundRecord // 历史记录（窗口=100轮）
}
```

**关键特性**：
- 历史窗口大小固定为 **100 轮**
- 每轮共识结束后通过 `UpdateScore()` 方法更新
- 用于动态验证者筛选（`ValidatorSelector` 根据 `minTrustScore` 过滤低质量节点）

> ⚠️ **注意**：Prepare、Commit 等阶段产生的大量临时投票消息并不必然全部作为链上数据长期保存。本文平台仅保留与**区块提交、状态恢复、信任评分更新和实验分析相关的必要数据**，以避免过度持久化带来额外存储开销。

## 3.3.2 系统数据存储设计

系统数据主要指实验平台在运行、监控、统计和分析过程中产生的辅助数据。与区块链数据不同，系统数据**不参与链上共识过程，也不影响交易排序和状态转移结果**，而是用于记录实验条件、运行状态和性能指标，从而保证实验过程可复现、实验结果可分析。

系统数据采用**双存储引擎架构**：
- **PostgreSQL**：用于 HCP-Loadgen 的结构化业务数据（Schema 隔离）
- **文件系统 (JSON/CSV)**：用于 HCP-Lab 的实验结果和日志数据

### 1. PostgreSQL 结构化存储 (HCP-Loadgen)

HCP-Loadgen 使用 PostgreSQL 数据库存储负载生成的业务数据，通过 **Schema 隔离机制**确保不同实验之间的数据独立性。

#### Schema 隔离机制

```rust
pub struct StorageConfig {
    pub database_url: String,           // 数据库连接URL
    pub db_schema: String,              // Schema名称（如 "exp1", "exp2"）
    pub reset_schema_on_start: bool,    // 是否在启动时重置Schema
    pub max_connections: u32,           // 连接池最大连接数
}
```

每次实验启动时，系统会自动创建独立的 Schema（如 `exp1_tpbft`、`exp2_hotstuff`），避免不同实验的数据相互干扰。

#### 数据表结构

##### (1) accounts 表 - 账户信息

```sql
CREATE TABLE {schema}.accounts (
    account_id BIGSERIAL PRIMARY KEY,
    address VARCHAR(64) NOT NULL UNIQUE,
    username VARCHAR(50),
    created_at TIMESTAMPTZ DEFAULT NOW()
);
```

##### (2) balances 表 - 余额快照

```sql
CREATE TABLE {schema}.balances (
    account_id BIGINT REFERENCES {schema}.accounts(account_id),
    asset_symbol VARCHAR(10) NOT NULL,
    available DECIMAL(20,8) DEFAULT 0,
    frozen DECIMAL(20,8) DEFAULT 0,
    updated_at TIMESTAMPTZ DEFAULT NOW(),
    PRIMARY KEY (account_id, asset_symbol)
);
```

##### (3) orders 表 - 订单信息

```sql
CREATE TABLE {schema}.orders (
    order_id BIGSERIAL PRIMARY KEY,
    account_id BIGINT REFERENCES {schema}.accounts(account_id),
    side VARCHAR(10) CHECK (side IN ('BUY', 'SELL')),
    price DECIMAL(20,8) NOT NULL,
    quantity DECIMAL(20,8) NOT NULL,
    filled_qty DECIMAL(20,8) DEFAULT 0,
    status VARCHAR(20) DEFAULT 'PENDING',
    timestamp TIMESTAMPTZ DEFAULT NOW()
);
```

##### (4) trades 表 - 交易记录

```sql
CREATE TABLE {schema}.trades (
    trade_id BIGSERIAL PRIMARY KEY,
    buy_order_id BIGINT,
    sell_order_id BIGINT,
    price DECIMAL(20,8) NOT NULL,
    quantity DECIMAL(20,8) NOT NULL,
    tx_hash VARCHAR(128),        -- 链上交易哈希
    latency_ms INTEGER,          -- 确认延迟（毫秒）
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 创建索引以加速查询
CREATE INDEX idx_{schema}_trades_tx_hash ON {schema}.trades(tx_hash);
CREATE INDEX idx_{schema}_balances_account_id ON {schema}.balances(account_id);
```

#### 技术优化

- **连接池管理**：使用 `PgPoolOptions` 管理数据库连接池，避免频繁创建/销毁连接
- **批量写入**：采用 PostgreSQL `Binary Copy In Writer` 进行批量写入，显著提升吞吐量
- **UPSERT 语义**：使用 `ON CONFLICT DO UPDATE` 实现幂等写入，防止重复数据

### 2. 文件系统存储 (HCP-Lab)

HCP-Lab 将实验结果以 **JSON 格式**保存到文件系统，同时输出 CSV 文件供后续分析使用。

#### 目录结构

```
report/
├── exp{N}_{algorithm}/               # 按实验编号和算法类型组织
│   ├── nodes_{count}/                 # 按节点规模分组
│   │   ├── result.json                # 性能指标（JSON格式）
│   │   ├── metrics.csv                # 详细数据（CSV格式）
│   │   └── logs/                      # 运行日志
│   └── figures/                       # 图表输出（PNG/SVG）
└── summary.json                       # 实验总览
```

#### JSON 结果结构

```json
{
  "name": "实验名称",
  "description": "实验描述",
  "points": [
    {
      "params": {
```

#### 关键指标说明

| 指标类别 | 指标名称 | 单位 | 说明 |
|----------|----------|------|------|
| **吞吐量** | tps | transactions/s | 每秒确认交易数 |
| **延迟** | avg_confirm_time_ms | ms | 平均确认延迟 |
| **延迟分位** | p50/p95/p99_ms | ms | 第50/95/99百分位延迟 |
| **资源利用** | cpu_percent | % | CPU利用率 |

| **存储性能** | rocksdb_write_avg_ms | ms | RocksDB平均写入延迟 |
| **存储性能** | rocksdb_write_amplification | ratio | 写入放大倍数 |
| **可靠性** | consensus_failures | count | 共识失败次数 |

### 3. 系统数据类型详解

#### (1) 实验配置数据 (Experiment Configuration)

用于描述每一轮实验的基本参数，由 HCP-Lab 的参数矩阵模块生成：

```python
params = {
    "nodes": [4, 8, 16, 32],           # 节点规模
    "tx": [1000, 5000, 10000],         # 交易数量
    "algorithm": ["tpbft", "hotstuff"], # 共识算法
    "repeat": 3,                        # 重复次数
    "warmup_time": 10,                  # 预热时间（秒）
    "random_seed": 42                   # 随机种子（保证可复现性）
}
```

通过保存实验配置数据，可以确保不同算法或不同参数组合在**相同条件下**运行，从而提高实验对比的公平性。

#### (2) 节点运行数据 (Node Monitoring Data)

用于记录实验过程中各节点的系统状态，由 SystemMonitor 模块实时采集：

```python
metrics = {
    "cpu_percent": 78.5,       # CPU利用率
    "mem_bytes": 2147483648,   # 内存占用（字节）
    "net_mbps": 1024.5,        # 网络吞吐（Mbps）
    "io_util": 45.2,           # 磁盘I/O利用率
    "io_await": 2.3,           # I/O等待时间（ms）
    "io_read_s": 102400000,    # 读取速率（bytes/s）
    "io_write_s": 51200000,    # 写入速率（bytes/s）
}
```

**用途**：当 P99 延迟明显上升时，可以结合 CPU 利用率、网络带宽和日志信息判断瓶颈来源：
- **签名验证瓶颈**：CPU 利用率高，但网络带宽低
- **消息广播瓶颈**：网络带宽高，CPU 利用率中等
- **状态持久化瓶颈**：磁盘 I/O 高，RocksDB 写入延迟大

#### (3) 性能指标数据 (Performance Metrics Data)

用于记录实验结果，是第四章进行基准对比、参数扫描、消融分析和性能边界建模的主要依据：

**采集方式**：HCP-Lab 在实验结束后通过解析以下时间戳计算延迟指标：
- 交易发送时间 (`tx_send_time`)
- 进入 Mempool 时间 (`mpool_insert_time`)
- 区块提交时间 (`block_commit_time`)
- 确认回执时间 (`confirm_receipt_time`)

**计算公式**：
```
端到端延迟 = confirm_receipt_time - tx_send_time
TPS = 总确认交易数 / 实验持续时间
P99延迟 = 排序后第99百分位的端到端延迟
```

#### (4) 日志与追踪数据 (Log & Trace Data)

用于记录实验过程中的关键事件，包括：

| 日志类型 | 解析器函数 | 提取指标 |
|----------|------------|----------|
| 区块提交日志 | `parse_block_times()` | 平均出块时间 |
| 交易确认日志 | `parse_confirm_times()` | 确认延迟分布 |
| RocksDB写入日志 | `parse_rocksdb_times()` | RocksDB写入延迟 |
| RocksDB WAL日志 | `parse_rocksdb_wal_synced()` | WAL同步时间 |
| RocksDB压缩日志 | `parse_rocksdb_compactions()` | Compaction耗时 |
| 共识失败日志 | `parse_consensus_failures()` | 失败次数 |

对于多轮自动化实验而言，**日志与统计结果的对应关系尤为重要**，因此本文平台在实验目录中按照**算法类型、节点规模、负载模式和实验轮次**组织结果文件，以便后续统一检索和分析。

---

## 总结

通过上述设计，系统数据在 HCP-Bench 中承担了**实验支撑层**的作用。它不改变链上共识过程，但能够完整记录实验运行环境、负载输入、系统状态和性能输出，从而保证实验结果具有可追溯性和可解释性。

**区块链数据与系统数据的分离**，使平台既能够保持共识执行路径的简洁性，又能够为性能分析提供充分的数据支撑。这种设计模式为后续章节的实验提供了坚实的数据基础。
