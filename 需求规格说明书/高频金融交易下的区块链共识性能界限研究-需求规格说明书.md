## **1\. 引言**

### **1.1 编写目的**

本需求规格说明书（Software Requirements Specification, SRS）用于明确毕业设计“高频金融交易下的区块链共识性能界限研究”的研究范围、系统组成、实验要求与验收标准。说明书的目标不是对外宣传系统能力，而是为课题实施提供可执行、可复现、可验证的工程依据。

结合当前项目现状，本课题已经形成由 `hcp-consensus`、`hcp-loadgen`、`hcp-lab` 以及根目录运维脚本共同构成的实验体系：`hcp-consensus` 负责共识引擎与状态机实现，`hcp-loadgen` 负责交易构造、签名与广播，`hcp-lab` 负责实验编排、指标采集与报告导出，`hcp` 根目录脚本负责节点启动与监控接入。后续需求应围绕这一现实架构展开，避免将尚未完成的机制描述为既成事实。

因此，本说明书重点回答三个问题：一是系统要测什么，即在高频交易负载下测量吞吐量、确认时延、尾延迟、扩展性与资源消耗；二是系统怎么测，即如何通过统一脚本、统一参数与统一采样口径完成多算法对照；三是结果如何判定，即如何以当前项目可达到的实验规模为基础，形成可信的阶段性性能边界结论。

### **1.2 背景与研究定位**

高频金融交易（High-Frequency Trading, HFT）对底层基础设施的核心要求并不是单一的“高 TPS”，而是在可接受的硬件成本下，尽量缩短交易确认时间、降低尾延迟波动，并在节点规模上升时维持基本可用的吞吐表现。对于联盟链或实验性许可链场景，这一问题通常落在 BFT 类协议及其优化路径上。

本课题的工程基础并非从零搭建，而是建立在现有 HCP 项目之上。当前项目已经支持 tPBFT、HotStuff、Raft、IBFT、PoW、Votor、分层共识、分层 tPBFT 等多种共识路线，具备比较不同协议在相同负载与相同容器环境下表现的条件。因此，本课题的现实定位是：以 tPBFT 及其相关优化为主线，以 HotStuff、IBFT、Votor 等实现为对照组，系统分析高频交易负载下链式 BFT 机制的性能边界。

在理论参考上，Sui Mysticeti、MonadBFT、AptosBFT、Sei Twin Turbo 等前沿系统仍然具有重要价值，但其作用应定义为“外部参照系”而非“本课题必须完整复现的工程目标”。这些协议展示了低延迟 DAG、流水线提交、快速恢复和网络优化等方向的可能上界，为本课题提供了分析视角；而本项目的主要任务，是在 Cosmos SDK / CometBFT 语境下，评估哪些思想可以被抽象、借鉴或用轻量实验近似验证。

此外，tPBFT 所依赖的信任评分思想与本课题关系最为紧密。针对高频场景中 PBFT 的通信放大和领导者瓶颈问题，相关研究通过高质量指标与低质量指标的规范化处理，分别采用
 $Y_{ij} = \frac{y_{ij} - y_{j,min}}{y_{j,max} - y_{j,min}}$
和
 $Y_{ik} = \frac{y_{k,max} - y_{ik}}{y_{k,max} - y_{k,min}}$
来衡量节点状态，并据此计算综合信任分值 $Q(n_i)$。这一路线与当前项目中的 `TrustScorer`、`ValidatorSelector` 等模块存在直接对应关系，因此比完全追求超高 TPS 的公开链宣传指标更适合作为本课题的研究落点。

### **1.3 术语定义**

为了减少研究与实现过程中的歧义，本说明书对核心术语作如下定义：

| 术语名称 | 英文全称 / 概念解释 | 课题语境下的特定工程含义 |
| :---- | :---- | :---- |
| **HFT** | High-Frequency Trading | 高频金融交易。在本课题中主要指由 `hcp-loadgen` 生成的高频转账或订单近似负载，强调突发性、时间敏感性和尾延迟约束。 |
| **TPS** | Transactions Per Second | 吞吐量，指系统单位时间内完成提交并被统计为成功事务的交易数。 |
| **共识延迟** | Consensus Latency | 交易从发出或被节点接收开始，到被系统判定为完成确认的时间间隔。研究中重点关注平均值、P95 与 P99。 |
| **Finality** | 确定性终局 | 交易在系统内达到不可逆确认的状态。对 BFT 类协议通常与提交证据或多数节点确认有关。 |
| **BFT** | Byzantine Fault Tolerance | 拜占庭容错。在总节点数满足 $N \geq 3f + 1$ 时，系统在不超过 $f$ 个恶意或失效节点的条件下仍保持安全性与活性。 |
| **DAG** | Directed Acyclic Graph | 有向无环图。本文中主要作为文献对照对象，用于分析与链式 BFT 在延迟和并行性上的差异。 |
| **Votor / Rotor** | \- | 受 Alpenglow 思想启发的实验性概念。其中 Votor 在本项目中主要体现为快速投票或快速确认路径；Rotor 更适合作为传播拓扑优化的参考概念，而非当前系统的完整既有实现。 |
| **QC / TC** | Quorum Certificate / Timeout Certificate | 法定人数证书与超时证书，主要用于 HotStuff / MonadBFT 一类协议的流程分析与故障恢复讨论。 |

## **2\. 课题目标与任务概述**

### **2.1 核心研究目标**

本课题的核心目标是构建一套面向毕业论文研究的区块链共识性能测试框架，并在统一环境下分析高频交易负载、节点规模与共识机制之间的关系。系统目标强调“可比较”和“可复现”，不以追求脱离当前项目基础的极端公开链指标为前提。

结合现有实现，研究重点放在以下三个方面：

1. 在 4、8、16、32 节点等典型实验规模下，对 tPBFT、HotStuff、IBFT、Votor、分层方案等进行控制变量测试。  
2. 在统一的负载模型下，对吞吐量、平均延迟、P99 延迟、资源消耗和节点扩展性进行阶段性量化。  
3. 在工程实现与文献分析之间建立联系，说明链式 BFT 方案在高频交易近似场景下的优势、瓶颈与可优化方向。

若实验资源允许，可继续向 64 节点扩展；若资源不足，则以 32 节点范围内形成高质量、可重复、结论清晰的边界曲线作为主要交付目标。

### **2.2 实验系统结构与模块分解**

当前实验系统由四部分组成：

1. **hcp-loadgen（负载生成层）**  
   该模块采用 Rust 与 Tokio 实现，负责交易构造、签名、广播、结果缓存与持久化回写。系统支持固定速率、突发、持续与抖动等多种发送模式，可通过 HTTP / gRPC 与被测节点交互。  
2. **hcp-consensus（共识与状态机层）**  
   该模块基于 Cosmos SDK 与 CometBFT 扩展实现，是研究主线所在。当前已接入 tPBFT、HotStuff、Raft、IBFT、PoW、Votor、分层共识等多种算法接口，支持在同一应用框架下进行横向比较。  
3. **hcp-lab（实验编排与分析层）**  
   该模块采用 Python 实现，负责参数矩阵展开、节点生命周期管理、日志解析、系统指标采集以及 Markdown / PDF / SVG 报告导出。  
4. **hcp 根目录脚本（运维与监控入口）**  
   包括 `start_nodes.sh`、`start_monitoring.sh`、`stop_hcp.sh` 等脚本，用于统一启动节点、配置监控与回收环境。

该结构说明本项目本质上是“统一实验平台 + 多共识对照组”的研究工程，而不是围绕单一算法的一次性演示。

### **2.3 实验运行环境规格**

#### **2.3.1 硬件基础设施环境**

为了保证实验结果具有可比性，建议使用如下基础环境：

* **计算资源**：多核 x86 服务器或实验室工作站，建议不少于 8 核 CPU。若进行 16 节点以上实验，建议 16 核及以上。  
* **内存资源**：建议 32GB 内存；小规模实验最低不低于 16GB。  
* **存储系统**：建议使用 SSD 或 NVMe SSD，以减轻日志写入与状态持久化对实验结果的干扰。  
* **网络条件**：以 1Gbps 及以上局域网或本机 Docker Bridge 为基础。若需要做网络扰动实验，可通过节流、丢包和时延注入模拟更复杂场景。

#### **2.3.2 软件与工具链环境**

* **操作系统**：Ubuntu 22.04 LTS。  
* **Go 环境**：Go 1.21+，用于 `hcp-consensus` 构建与运行。  
* **Rust 环境**：Rust 1.75+，用于 `hcp-loadgen` 构建与运行。  
* **Python 环境**：Python 3.10+，用于 `hcp-lab` 的实验编排与分析。  
* **容器与监控**：Docker Engine 24.x、Docker Compose V2、Prometheus；需要时可配合 iostat、LaTeX 工具链导出报告。  
* **数据库组件**：PostgreSQL 为可选实验组件，主要用于负载生成器结果持久化与隔离。

### **2.4 关键核心技术拆解**

1. **多共识引擎统一接入**  
   通过统一接口在同一实验框架内接入 tPBFT、HotStuff、IBFT、Raft、Votor 等算法，降低横向比较时的环境偏差。  
2. **高频负载生成与冷热路径分离**  
   利用 Rust 异步运行时、账户池和批量持久化机制，尽量减少数据库写入对发送热路径的影响。  
3. **信任评分与动态验证者选择**  
   以 tPBFT 为主线，研究信任评分是否能够缓解 PBFT 在高频场景中的广播与节点选择开销。  
4. **实验自动化与报告自动导出**  
   通过 `hcp-lab` 的参数矩阵、日志解析、图表导出与批量运行机制，保证实验具备重复执行能力。  
5. **故障与恢复分析**  
   在可控范围内模拟节点停机、异常退出或网络波动，观察不同协议对活性与时延的影响。

## **3\. 研究/实验需求说明**

### **3.1 数据需求规格（Data Specification）**

高频交易场景下，共识性能评估必须建立在结构化、可追踪的数据之上。当前系统的数据需求定义如下：

* **输入数据流**  
  `hcp-loadgen` 负责生成交易数据。交易可以采用 JSON 或 Protobuf 等便于工程实现的格式，但至少应包含以下字段：  
  * `timestamp_in`：交易生成或发送时间戳；  
  * `tx_hash`：交易唯一标识；  
  * `signature`：Ed25519 等签名字段，用于保证验签成本真实存在；  
  * `payload_data`：转账、挂单、撤单或其他近似 HFT 负载内容。  
* **并发与负载规模**  
  单轮实验以 $10^3$ 至 $10^5$ 级交易规模为主，支持 `fixed`、`burst`、`sustained`、`jitter` 等发送模式。若资源允许，可逐步提高到更大规模，但不将百万级负载设为所有实验的前置要求。  
* **输出数据规范**  
  系统应输出结构化日志与 CSV/JSON 结果，至少包含：`Tx_ID`、`Node_ID`、`Block_Height`、`Submit_Time`、`Commit_Time`、`Latency`、`Success_Flag` 等字段。对支持阶段化统计的协议，可附加 `Propose_Time`、`Prevote_Time`、`Precommit_Time` 等字段。

### **3.2 算法实验功能模块需求**

#### **3.2.1 高频性能遥测与状态机追踪模块**

**功能描述**：记录交易从进入系统到完成确认的关键时间点，用于计算平均时延、P95、P99、区块时间与确认路径差异。

**技术实现标准**：

* 在可插桩的共识阶段记录关键时序数据，避免仅依赖单一日志时间。  
* 对 Votor、IBFT、HotStuff 等不同协议保留协议相关字段，便于事后对比。  
* 对无法完整实现的外部协议，仅允许以理论或模型参数方式对照，不得与实测数据混写。

#### **3.2.2 系统资源探针与通信瓶颈分析模块**

**功能描述**：从实验平台外部持续采集 CPU、内存、网络与 I/O 指标，用于识别性能瓶颈。

**技术实现标准**：

* 记录节点级 CPU、内存、网络收发与磁盘写入情况。  
* 结合日志统计消息量与吞吐变化，分析不同协议在节点扩展时的通信成本趋势。  
* 在报告中明确区分“共识瓶颈”“网络瓶颈”“存储瓶颈”和“负载生成端瓶颈”。

#### **3.2.3 共识切换与图表对比模块**

**功能描述**：支持在统一实验脚本中切换共识算法与参数，并自动输出对比图表。

**技术实现标准**：

* 参数通过 JSON、YAML 或脚本参数统一注入。  
* 每次实验结束后自动生成 TPS 曲线、延迟分布图、节点规模对比图等基础结果。  
* 对标外部协议时，应将“实测结果”“模拟结果”“文献引用结果”分别标识。

#### **3.2.4 故障注入与异常恢复验证模块**

**功能描述**：验证系统在节点退出、网络抖动或领导者异常情况下的降级与恢复能力。

**技术实现标准**：

* 支持通过容器控制或脚本方式模拟节点停机、重启和网络中断。  
* 记录故障发生后系统恢复出块、恢复提交或重新稳定所需的时间。  
* 对结果的解释应以“实验观察”为主，不夸大为完整拜占庭攻击验证。

### **3.3 非功能需求规格说明**

#### **3.3.1 可扩展性**

系统需支持从 4 节点逐步扩展到 32 节点的标准实验规模，并在资源允许时进一步尝试 64 节点。扩展性评估重点不是证明系统在超大规模下仍保持理想性能，而是识别性能下降开始出现的节点区间，并解释其原因。

#### **3.3.2 容错性与数据采集连续性**

在部分节点异常退出或单轮实验失败时，平台应尽量保证采集程序和其余节点日志仍可被保留，避免一次失败导致整组实验数据丢失。只要健康节点数量满足基本容错假设，系统应继续观察而不是直接中止全部实验。

#### **3.3.3 密码学真实性**

负载生成与节点处理过程中应保留真实签名与验签步骤，避免以“空交易”或省略签名计算的方式获得不具有解释力的高吞吐结果。若某轮实验关闭验签或采用简化模式，必须在报告中显式说明。

## **4\. 性能评价指标（核心验证标尺）**

### **4.1 分阶段性能目标**

结合当前项目已有实验结果，本课题采用“分阶段、分场景”评价，而不是设定脱离现有实现基础的单一极端目标。

| 指标维度 | 标准化测试条件 | 阶段性目标 | 说明 |
| :---- | :---- | :---- | :---- |
| **基线吞吐能力** | 4 节点，本地 Docker/局域网，统一负载与脚本 | tPBFT 基线达到或接近 50 tx/s 以上 | 与当前项目已有 4 节点结果保持同一量级，用于证明平台与主线算法可用 |
| **低延迟优化能力** | 16 或 32 节点，对 Votor、IBFT、分层方案等进行对照 | 在特定优化方案下将 P99 或 finalize 指标压低至 150ms 左右或更好 | 该指标用于体现结构优化价值，不要求所有算法同时满足 |
| **扩展性观察能力** | 节点规模从 4 扩展至 32 | 吞吐随节点扩展出现可解释的衰减曲线，并保持系统持续可运行 | 重点是形成边界分析，不强行要求 4 到 32 节点几乎无损扩展 |
| **结果可复现性** | 同一参数下重复实验不少于 5 次 | 主要指标具备稳定趋势，能计算 95% 置信区间 | 可复现性优先于单次峰值成绩 |

### **4.2 实验可靠性保障与统计学严谨性约束**

* 同一实验点原则上重复执行不少于 5 次。  
* 统计时应剔除冷启动预热数据与实验结束前的异常尾段数据。  
* 结果报告至少给出平均值、P95、P99 与必要的标准差或置信区间。  
* 若采用文献数据作对照，应与本地实测结果分栏呈现，不得混淆来源。

## **5\. 约束与进度要求**

### **5.1 技术与框架约束边界**

1. **框架继承约束**：核心实验平台以 Cosmos SDK / CometBFT 为底座，Rust 负载生成器与 Python 编排器为外部协同组件。课题重点是基于现有框架开展实验与优化，而非重写一套全新的公链内核。  
2. **对照公平性约束**：不同算法的比较必须在尽量一致的节点规模、硬件限制、负载模型与采样口径下进行。  
3. **表述边界约束**：外部协议如 Mysticeti、MonadBFT、AptosBFT 等主要用于理论参照或模型映射，除非已有可运行实现，否则不将其写作本项目已完成模块。
